# DDD Open Questions

Purpose: targeted questions whose answers would change implementation, compatibility, documentation, or future agent behavior.

| Question | Why It Matters | Affected Contexts | Evidence | Needed From | Status |
|---|---|---|---|---|---|
| Which `--json` command outputs are stable machine-readable contracts? | Agents need to know whether JSON fields can be renamed or reshaped. | CLI, Control, Harness, Local Store | Global `--json` in `src/ces/cli/__init__.py`; JSON assertions in `tests/unit/test_cli/`; generated reports/status paths. | Product / Code Owner | Open |
| Should `ces run` remain a permanent alias for `ces build`? | Removing it would break older docs, habits, or automation. | CLI, Docs | `src/ces/cli/__init__.py` registers `run` as "Legacy alias"; current docs prefer `build`. | Product Owner | Open |
| Is the persisted evidence dict shape an official JSON contract or an implementation detail? | Evidence changes affect report/status/review consumers and historical `.ces/state.db` reads. | Harness, Control, Local Store, CLI | `EvidencePacket` exists, but `LocalProjectStore.save_evidence` stores dict content. | Code Owner | Open |
| Should local SQLite schema changes get an explicit schema version or migration ledger? | Future agents need a safe pattern for changing `.ces/state.db` without data loss. | Local Store, Control, Brownfield, Harness | `LocalProjectStore.initialize()` includes in-app migrations but no obvious schema version table. | Code Owner | Open |
| Should brownfield docs use `change/retire`, keep `migrate/remove` as aliases, or expose both with different meanings? | Code enum values and docs terminology diverge; CLI compatibility may depend on accepted values. | Brownfield, CLI, Docs, Local Store | `LegacyDisposition` uses `preserve/change/retire/under_investigation/new`; `docs/Brownfield_Guide.md` says `preserve/migrate/remove`. | Product / Code Owner | Open |
| Which expert workflow commands are long-term public surface versus compatibility surface? | Some commands are central; others are older or excluded from coverage gates. | CLI, Docs, Tests | `README.md` lists many advanced commands; `pyproject.toml` coverage omit comments mention older expert commands. | Product Owner | Open |
| Is `ces spec` stable enough to treat spec Markdown as a public file format? | Parser/model changes could break existing specs or generated manifests. | Spec Authoring, Control, CLI | `SpecDocument`, `SpecParser`, default templates, fixtures, property tests, `ces build --from-spec` tests. | Product / Code Owner | Open |
| Should `TaskManifest` be described as a truth artifact or a separate governed operational contract? | It inherits `GovernedArtifactBase`, but the `TruthArtifact` union excludes it. | Control Plane, Contracts, Docs | `src/ces/control/models/manifest.py`; `src/ces/control/models/__init__.py`. | Domain Expert / Code Owner | Open |
| What is the support/deprecation plan for compatibility Alembic migrations under tests? | Agents may misread root `alembic.ini` and test revisions as active production migrations. | Compatibility Tests, Local Store, Docs | `alembic.ini` points to `tests/integration/_compat/alembic`; docs say local SQLite is supported runtime. | Code Owner | Open |
| Which metric names and Grafana dashboard fields are stable? | External dashboards may depend on names even if observability is optional. | Observability, Examples, Docs | `src/ces/observability/`; `examples/grafana-*.json`; optional observability extra. | Code Owner | Open |
| What runtime version/auth matrix is supported for Codex and Claude adapters? | External CLIs change outside CES; safe execution and evidence behavior may drift. | Execution, CLI, Docs | `RuntimeRegistry` detects `codex`/`claude`; docs require a supported local runtime but do not pin versions. | Code Owner | Open |
| Should completion claim stdout blocks be versioned and documented for external agents? | Parser changes can break evidence extraction from agent output. | Execution, Harness, Contracts | `src/ces/execution/completion_parser.py`; `src/ces/harness/models/completion_claim.py`. | Code Owner | Open |
| Are generated CI templates expected to stay backward compatible across minor versions? | Tightening `ces setup-ci` templates can break user repositories. | CLI, Packaging, CI | `src/ces/cli/templates/ci/`; `src/ces/cli/setup_ci_cmd.py`; setup-ci tests. | Product / Release Owner | Open |
| Should starter manifest templates be strict `TaskManifest` examples or human-editable sketches? | Template fields such as `tier_c` differ from enum wire values; agents may use them as strict schemas. | CLI, Control, Docs | `src/ces/cli/templates/manifests/*.yaml`; `TaskManifest` enum values. | Product / Code Owner | Open |
| Should package-level Python imports be treated as public library API or internal convenience? | Removing exports may break unknown users if CES is imported as a library. | Packaging, Control, Harness, Execution | Package `__init__.py` files and public export tests. | Product Owner | Open |
| What is the intended boundary for examples importing compatibility test modules? | `examples/freshcart/seed_data.py` imports `tests.integration._compat.*`; future agents may copy test-only dependencies into product code. | Examples, Compatibility Tests, Packaging | Cross-context scan; example seed data tests. | Code Owner | Open |
| Are historical PRD/Implementation/Roadmap docs allowed to drive new code, or only explain design history? | Future agents may revive FastAPI, Celery, Redis, or Postgres work against the current local-first contract. | Docs, CLI, Packaging, Local Store | Historical docs are labeled historical; `reject_server_mode` rejects server mode. | Product Owner | Open |
