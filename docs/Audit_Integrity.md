# Audit Integrity

CES records governance events in a project-local append-only audit ledger stored in `.ces/state.db`.

## Integrity Model

Each audit entry contains:

- stable event fields such as event type, actor, action summary, decision, rationale, scope, and project id
- `prev_hash`, linking the entry to the previous entry for the same project
- `entry_hash`, an HMAC-SHA256 digest over the entry data and previous hash

The first entry links to `GENESIS`. Each later entry links to the prior entry's `entry_hash`.

The HMAC secret is generated by `ces init` at `.ces/keys/audit.hmac`, stored mode `0600`, and should not be committed or reused across projects.

## Verify the Ledger

Run:

```bash
ces audit verify
```

Machine-readable form:

```bash
ces --json audit verify
```

Expected success payload:

```json
{
  "project_id": "local-project",
  "verified": true,
  "status": "valid"
}
```

If verification fails, CES exits nonzero and reports `tampered_or_invalid`. Treat that as an incident-response signal: stop relying on the affected audit history until you have investigated direct database edits, wrong keys, restored stale state, or partial backups.

## Key Rotation

Rotate audit keys only at a clear operational boundary:

1. Export or archive current proof/audit artifacts if they matter.
2. Back up `.ces/state.db` and `.ces/keys/`.
3. Remove `.ces/keys/audit.hmac`.
4. Run `ces doctor --security` or `ces init` to regenerate local key material as appropriate.
5. Record the rotation in project notes.

Old entries cannot be verified with the new HMAC secret. Preserve the old secret with the archived state if historical verification matters.

## Tamper Cases CES Detects

`ces audit verify` fails when:

- an entry's content changes without recomputing the HMAC
- an entry's `prev_hash` points at the wrong predecessor
- entries are reordered
- verification uses the wrong HMAC secret
- restored state is incomplete or inconsistent

SQLite triggers also reject normal update/delete attempts against `audit_entries`, but direct filesystem or database tampering can still corrupt the file. The HMAC chain is the detection layer.
